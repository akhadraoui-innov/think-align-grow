# v2.8 — Lot 8 — Brique E : Rate limiting opérations sensibles

**Date** : 27 avril 2026
**Périmètre** : sécurité opérationnelle, hardening edge functions admin

## Contexte

Deux edge functions admin (`delete-user` et `impersonate-user`) protégées par garde `super_admin` mais sans contre-mesure contre l'abus (token compromis, script malveillant) : un attaquant ayant capturé un JWT super_admin pouvait théoriquement supprimer des centaines de comptes en quelques secondes.

## Livrable

### Migration SQL (`20260427064815_*.sql`)

- **Table `public.rate_limits`** : `(user_id, action_key, window_start, count)` — RLS activée, **AUCUNE policy** → accessible uniquement par `service_role`.
- **Index unique** `(user_id, action_key, window_start)` pour permettre l'upsert atomique.
- **Fonction `check_rate_limit`** SECURITY DEFINER, `REVOKE EXECUTE` de PUBLIC/anon/authenticated → callable uniquement par service_role. Implémente une fenêtre glissante alignée et retourne `{ allowed, count, limit, remaining|reset_at }`.
- **Fonction `cleanup_rate_limits`** : DELETE des entrées > 24h, à cron-er.

### Edge functions équipées

| Fonction | Limite | Réponse en cas de dépassement |
|---|---|---|
| `delete-user` | **5 / 60 min** par super_admin | `429 rate_limit_exceeded` avec `reset_at` |
| `impersonate-user` | **10 / 60 min** par super_admin | `429 rate_limit_exceeded` avec `reset_at` |

Le contrôle est appliqué **après** la garde `is_super_admin` pour ne pas comptabiliser les requêtes non autorisées.
