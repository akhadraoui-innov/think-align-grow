# v2.7 — Lot 7 — Hardening EXECUTE permissions sur SECURITY DEFINER

**Date** : 27 avril 2026
**Périmètre** : sécurité base de données, permissions fonctions

## Contexte

Avant ce lot, le scanner Supabase remontait **165 findings** (4 errors + 155 warnings + 6 info) :
- 4 errors Vault (secrets en clair) déclassés en `warn` après audit RLS au Lot 6.
- 155 warnings de type `*_security_definer_function_executable` : toute fonction SECURITY DEFINER était exécutable par défaut par `PUBLIC`, `anon`, `authenticated` — y compris les triggers internes et helpers `service_role`.

## Migration livrée

`supabase/migrations/20260427061828_*.sql` :

1. **REVOKE global** : `REVOKE EXECUTE ON FUNCTION` sur les **77 fonctions SECURITY DEFINER** du schéma `public`, retiré de `PUBLIC`, `anon`, `authenticated`.
2. **Whitelist `anon` (5 fonctions)** :
   - `validate_confirmation_token`, `validate_unsubscribe_token`, `consume_unsubscribe_token` (liens email transactionnels)
   - `get_invitation_by_token` (acceptation d'invitation pré-login)
   - `find_workshop_by_code` (rejoindre un workshop par code public)
3. **Whitelist `authenticated` (~36 fonctions)** : helpers RLS (`has_role`, `is_org_admin`, `is_saas_team`, `is_workshop_*`, `has_academy_access`, `requires_2fa`, `is_url_allowed`), notifications (`mark_*_read`), quotas (`check_*_quota`, `get_org_effective_features`), prompts UCM, RGPD (`export/erase_user_email_data`), opt-in (`confirm_email_opt_in`), métriques admin avec garde interne `is_saas_team` (`get_email_*`, `get_edge_function_metrics`, `get_priority_lane_metrics`, `get_system_health`, `count_users_by_role`, `verify_audit_chain_integrity`, `review_email_security_flag`, `replay_dlq_message`).
4. **Triggers et helpers service_role** : aucun GRANT — exécutables uniquement par owner postgres ou via `service_role` bypass.

## Findings résiduels acceptés

Les **41 warnings** restants sont **par design** : le scanner Supabase signale TOUTE fonction SECURITY DEFINER appelable par un rôle anon/authenticated, sans distinguer les usages légitimes (helpers RLS, RPCs publiques, garde interne) des fuites réelles. Pour ces 41 fonctions :
- 5 = whitelist `anon` (liens email + invitations + workshop par code).
- ~36 = whitelist `authenticated` (helpers RLS et opérations client garde-protégées).
- 1 = `REALTIME_CHANNEL_UNAUTHORIZED_SUBSCRIPTION` mitigé par règle "Postgres Changes only" documentée dans `data-protection-baseline.md`.
