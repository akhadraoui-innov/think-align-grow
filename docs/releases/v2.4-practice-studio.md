# Release Note — Practice Studio v2.4

**Date** : 16 avril 2026
**Périmètre** : Admin Practice Studio · Portail apprenant · Cabinet Simulator
**Statut** : ✅ Livré en production

---

## 🎯 Objectifs de la release

Aligner les trois surfaces (Admin / Portail / Cabinet) sur un référentiel unique de pratiques, sécuriser la bibliothèque de blocs réutilisables en multi-tenant, et fluidifier l'expérience auteur du Practice Studio.

---

## ✨ Nouveautés

### 1. Practice Studio (Admin)

#### Distribution & Assignations — corrigé
- **Fix critique** : la liste des apprenants assignables s'affichait vide. La requête `profiles` utilise désormais `user_id` + `display_name` (au lieu de `id`), et les assignations sont créées avec la bonne clé étrangère, garantissant l'application des politiques RLS pour les apprenants.

#### Identité — auto-suggestion
- Le champ **Univers** se pré-remplit automatiquement quand l'auteur change le **type de pratique**, en s'appuyant sur le `MODE_REGISTRY`. La valeur reste librement modifiable.

#### Topbar — autosave only
- **Suppression du bouton "Sauver"** du topbar. L'autosave (debounce 800 ms) et le raccourci ⌘S restent les seuls mécanismes de persistance, ce qui élimine les races conditions et clarifie l'UX.

#### Bibliothèque de blocs — sécurité multi-tenant
Refonte complète du modèle de visibilité :

| Aspect | Comportement v2.4 |
|---|---|
| **Création par défaut** | `is_global = false` → bloc visible uniquement dans l'organisation du créateur |
| **Filtres scope** | Trois onglets : *Tous · Mes blocs · Globaux* |
| **Promotion en global** | Réservée aux **super-admins** (bouton 👑 *Promouvoir*) |
| **Suppression** | Auteur du bloc OU super-admin |
| **Badges** | Chaque bloc affiche son scope (🌍 Global / 🏢 Org) |

> Cette règle aligne la bibliothèque sur l'architecture multi-tenant existante (`OrgProvider` + RLS par `organization_id`).

#### Analytics — granularité fine
- **Score moyen par dimension** : nouveau graphique en barres agrégeant `evaluation.dimensions[].score` sur l'ensemble des sessions complétées.
- **Seuil de désignation du gagnant A/B** : le trophée n'apparaît plus que si chaque variante éligible cumule **≥ 3 sessions**. Un badge *« Données insuffisantes »* est affiché en attendant.

---

### 2. Portail apprenant

#### Catalogue de pratiques (`/portal/pratique`)
- **Badges réels** : chaque pratique affiche son origine (🌐 *Public* / 🏢 *Org* / ✅ *Assignée*) calculée en croisant `practice_organizations` et `practice_user_assignments`.
- **Échéances visibles** : les pratiques assignées affichent leur date limite avec un compteur `J-X` en rouge dès qu'il reste **3 jours ou moins**.

#### Dashboard formations (`/portal/formations/dashboard`)
- **Nouvelle section « Pratiques assignées »** : carte dédiée listant toutes les pratiques personnellement assignées à l'apprenant, triées par échéance ascendante avec alerte visuelle J-3.

---

### 3. Cabinet Simulator (`/simulator`)

- **Alignement complet** sur la logique du portail : le cabinet charge désormais l'union des pratiques visibles (publiques + ciblées org + assignées personnelles) via la même requête RLS.
- **Catalogue unifié** : les pratiques admin sont fusionnées avec les modes génériques `MODE_REGISTRY` dans un sélecteur unique.

### Modèle de données
Aucune migration DB. Les colonnes utilisées (`practice_organizations`, `practice_user_assignments`, `practice_blocks.is_global`, `academy_practices.organization_id`) existaient déjà.

### RLS — règles appliquées
- `academy_practices` : visibilité = `is_public = true` OR `organization_id IN (mes orgs)` OR `id IN (assignments where user_id = auth.uid())`.
- `practice_blocks` : visibilité = `is_global = true` OR `organization_id IN (mes orgs)`.
- `profiles` : SELECT autorisé pour les membres de la même org (utilisé par le picker d'assignation).
